# RC-022 应用关闭时自动保存会话 & 启动时自动恢复

## 1. 需求概述

| 项目 | 内容 |
|------|------|
| 需求编号 | RC-022 |
| 需求名称 | 应用关闭时自动保存会话 & 启动时自动恢复 |
| 提出日期 | 2026-05-09 |
| 优先级 | P1-高 |
| 影响范围 | 全栈（主进程 + 渲染进程） |

### 1.1 背景

当前应用关闭时的行为：
- 如果有未保存的文件，会逐个弹出"是否保存"对话框，用户需要逐个确认
- 关闭后所有打开的 Tab 状态全部丢失
- 下次启动时只会创建一个空白 Tab，之前打开的所有文件需要手动逐一重新打开
- 每个 Tab 之前选择的数据库连接和数据库也需要手动重新选择

这在日常使用中非常不便，尤其是同时打开多个查询文件、分别连接不同数据库的场景。

### 1.2 目标

实现类似 VS Code 的"热退出"（Hot Exit）机制：

1. **关闭时**：不再弹出保存确认对话框，直接关闭，所有 Tab 的状态（包括未保存的内容）自动缓存到本地
2. **启动时**：自动恢复上次关闭时的所有 Tab，包括 SQL 内容、未保存状态、数据库连接选择等
3. **连接恢复**：每个 Tab 关联的数据库连接和数据库名自动恢复到 UI 上（连接本身按需建立）

---

## 2. 需求详情

### 2.1 功能一：关闭时自动保存会话状态

#### 2.1.1 功能描述

应用关闭时（点击关闭按钮 / Alt+F4 / 系统关机），**跳过保存确认对话框**，自动将当前所有 Tab 的完整状态序列化并持久化到本地文件。

#### 2.1.2 需要保存的数据

每个 Tab 保存以下信息：

| 字段 | 说明 | 必须 |
|------|------|------|
| `id` | Tab 唯一标识 | 是 |
| `title` | Tab 显示标题 | 是 |
| `baseTitle` | 基础标题（如"查询 1"） | 是 |
| `content` | SQL 文本内容（**含未保存的修改**） | 是 |
| `filePath` | 关联的文件路径（无文件则为空） | 否 |
| `isDirty` | 是否有未保存修改 | 是 |
| `connectionId` | 关联的数据库连接 ID | 否 |
| `databaseName` | 选中的数据库名 | 否 |
| `maxRows` | 最大返回行数设置 | 是 |

全局状态：

| 字段 | 说明 |
|------|------|
| `activeTabId` | 当前激活的 Tab ID |
| `tabCounter` | Tab 计数器（避免恢复后标题编号重复） |

#### 2.1.3 存储位置与格式

- 存储路径：`{app.getPath('userData')}/config/session-state.json`
- 存储格式：JSON
- 遵循项目已有的持久化模式（参考 `settings-store.ts`、`connection-store.ts`）

#### 2.1.4 关闭行为变更

| 场景 | 当前行为 | 变更后行为 |
|------|---------|-----------|
| 所有 Tab 已保存 | 直接关闭 | 保存会话状态 → 直接关闭 |
| 部分 Tab 未保存 | 逐个弹出保存对话框 | 保存会话状态（含未保存内容）→ **直接关闭，不弹对话框** |
| 全部 Tab 未保存 | 逐个弹出保存对话框 | 保存会话状态（含未保存内容）→ **直接关闭，不弹对话框** |

#### 2.1.5 自动保存时机

除了关闭时保存，还应在以下时机**增量保存**，防止意外崩溃导致丢失：

| 时机 | 说明 |
|------|------|
| 关闭窗口时 | `mainWindow.on('close')` 事件中保存（必须） |
| Tab 内容变更时 | 防抖保存（建议间隔 5 秒），防止崩溃丢失（可选，建议实现） |
| Tab 增删 / 切换连接时 | 状态变化后立即保存（可选，建议实现） |

---

### 2.2 功能二：启动时自动恢复会话

#### 2.2.1 功能描述

应用启动时，读取上次保存的会话状态文件，恢复所有 Tab 到关闭前的状态。

#### 2.2.2 恢复内容

| 恢复项 | 描述 |
|--------|------|
| Tab 列表 | 按保存时的顺序恢复所有 Tab |
| Tab 内容 | 恢复每个 Tab 的 SQL 文本内容 |
| 未保存状态 | 如果关闭时 `isDirty=true`，恢复后仍标记为未保存（Tab 标题显示修改标记） |
| 当前激活 Tab | 恢复到关闭前激活的 Tab |
| 连接选择 | 恢复每个 Tab 的 `connectionId` 和 `databaseName` 到 UI 下拉框 |
| 最大行数 | 恢复每个 Tab 的 `maxRows` 设置 |

#### 2.2.3 恢复策略

| 场景 | 处理方式 |
|------|----------|
| 正常恢复 | 读取 `session-state.json`，按保存的数据重建所有 Tab |
| 会话文件不存在（首次启动） | 创建一个默认空白 Tab（保持当前行为） |
| 会话文件损坏 / 解析失败 | 记录日志，创建一个默认空白 Tab |
| 保存的 Tab 列表为空 | 创建一个默认空白 Tab |
| `connectionId` 对应的连接已被删除 | Tab 正常恢复但连接下拉框显示为空（不报错） |
| `filePath` 指向的文件已被外部删除 | Tab 正常恢复内容，保持 `isDirty=true`，`filePath` 保留（下次保存时用户可选择新路径） |
| `filePath` 指向的文件被外部修改 | Tab 恢复缓存的内容（关闭时的版本），标记为 `isDirty=true` |

#### 2.2.4 连接恢复策略

Tab 恢复时，**仅恢复 UI 上的连接和数据库选择**，不自动建立物理连接。用户执行 SQL 时才按需创建物理连接（复用现有的 `ensureSession` 机制）。

| 项目 | 行为 |
|------|------|
| 连接下拉框 | 恢复显示为之前选择的连接名称 |
| 数据库下拉框 | 恢复显示为之前选择的数据库名 |
| 物理连接 | **不自动建立**，用户执行 SQL 时按需建立 |
| 连接已被删除 | 下拉框显示为空，用户需重新选择 |

---

### 2.3 功能三：有文件路径的 Tab 的特殊处理

#### 2.3.1 有文件路径且未修改（`isDirty=false`）

- 会话缓存中**仍保存内容**（避免外部文件变动导致恢复为空）
- 恢复时加载缓存内容，标记为未修改状态

#### 2.3.2 有文件路径且已修改（`isDirty=true`）

- 会话缓存中保存的是**用户编辑后的内容**（而非磁盘上的文件内容）
- 恢复时加载缓存内容，标记为已修改（`isDirty=true`）
- 用户可以选择 Ctrl+S 保存到文件，或 Ctrl+Z 撤销修改

#### 2.3.3 无文件路径的 Tab（新建未保存）

- 内容完整保存到会话缓存
- 恢复后仍为无文件路径状态，标记为 `isDirty=true`（只要有内容）

## 3. 交互设计

### 3.1 关闭时的用户体验

**变更前：**
```
用户点击关闭 → 弹出"查询1未保存，是否保存？" → 用户选择 → 弹出"查询2未保存..." → ... → 关闭
```

**变更后：**
```
用户点击关闭 → 直接关闭（后台自动保存会话状态）
```

### 3.2 启动时的用户体验

**变更前：**
```
启动 → 显示一个空白 Tab
```

**变更后：**
```
启动 → 恢复上次关闭时的所有 Tab（含内容、连接选择、未保存标记）
```

### 3.3 未保存标记的视觉表现

恢复后，未保存的 Tab 仍然显示现有的修改标记（Tab 标题旁的圆点或星号），与正常编辑时的行为一致。

## 5. 影响分析

### 5.1 受影响的模块

| 模块 | 文件路径 | 修改类型 | 说明 |
|------|----------|----------|------|
| 会话状态存储 | `src/main/storage/session-state-store.ts` | **新增** | 会话状态的读写服务，参考 `settings-store.ts` 模式 |
| 主进程入口 | `src/main/index.ts` | 修改 | 关闭事件中调用会话保存；新增 IPC 通道 |
| IPC 通道常量 | `src/shared/constants/index.ts` | 修改 | 新增会话相关 IPC channel |
| IPC 处理 | `src/main/ipc/` | 修改 | 新增会话保存/恢复的 IPC handler |
| Preload 桥接 | `src/preload/index.ts` | 修改 | 暴露会话相关 API 给渲染进程 |
| 编辑器 Store | `src/renderer/stores/editor.ts` | 修改 | `init()` 方法改为从会话恢复；新增序列化/反序列化方法 |
| App 根组件 | `src/renderer/App.vue` | 修改 | `handleBeforeClose()` 移除保存确认逻辑，改为发送会话数据 |

- 依赖现有的连接配置持久化（`connection-store.ts`）——恢复时通过 `connectionId` 查找连接
- 依赖现有的 `EditorTab` 数据结构——序列化/反序列化的基础
- 无外部依赖，不需要新增第三方库

## 7. 非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 关闭时会话保存应在 500ms 内完成，不影响关闭体验 |
| 性能 | 启动时恢复应在 1s 内完成（20 个 Tab 以内） |
| 可靠性 | 会话文件写入采用原子写入（先写临时文件再重命名），避免写入中断导致文件损坏 |
| 可靠性 | 增量保存采用防抖策略（5 秒间隔），避免频繁 I/O |
| 容错性 | 会话文件不存在或损坏时，静默降级为创建空白 Tab，不报错 |
| 存储 | 会话文件大小应控制在合理范围（对超大 SQL 内容可考虑压缩或单独存储） |
| 安全性 | 会话文件中不包含密码等敏感信息（`connectionId` 仅为引用，密码在 `connections.dat` 中加密存储） |

## 8. 技术可行性评估

### 8.1 评估结论：✅ 完全可行

| 维度 | 评估 | 说明 |
|------|------|------|
| Tab 数据可序列化 | ✅ | `EditorTab` 是纯数据对象，直接 JSON 序列化 |
| 关闭时机可拦截 | ✅ | 已有 `mainWindow.on('close', e.preventDefault())` 机制 |
| 持久化基础设施 | ✅ | 已有 `app.getPath('userData')/config/` + JSON 读写模式 |
| 连接引用可恢复 | ✅ | `connectionId` 引用的连接配置已持久化在 `connections.dat` |
| Tab 内容可恢复 | ✅ | 内容在 `content` 字段中，无需额外处理 |
| 按需连接 | ✅ | 已有 `ensureSession` 机制，执行 SQL 时自动建立连接 |

### 8.2 风险点

| 风险 | 影响 | 应对策略 |
|------|------|----------|
| 应用崩溃导致会话丢失 | 中 | 实现增量保存（防抖 5 秒），减少丢失窗口 |
| 超大 SQL 导致保存/恢复慢 | 低 | 大多数场景 SQL 不超过几百 KB，后续可对大内容单独存文件 |
| 关闭单个 Tab 的保存行为 | 低 | 热退出仅在关闭应用时生效，单个 Tab 关闭仍走现有确认流程 |

- 本功能参考 VS Code 的 Hot Exit 机制（`"files.hotExit": "onExit"`）
