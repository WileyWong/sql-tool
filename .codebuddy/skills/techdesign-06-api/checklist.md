# API接口设计检查清单

## 设计完整性
- [ ] RESTful规范：路径用名词复数，HTTP方法正确
- [ ] 资源识别完整，路径层级清晰
- [ ] CRUD操作完整（创建/查询/更新/删除）
- [ ] 列表接口支持分页、过滤、排序

## 数据模型
- [ ] 统一响应格式（code/message/data）
- [ ] 请求模型有验证规则
- [ ] 响应模型无敏感信息
- [ ] 数据类型选择正确（金额用String，日期用ISO 8601）
- [ ] 枚举值使用大写+下划线（如 ORDER_STATUS）

## 错误处理
- [ ] HTTP状态码使用正确（200成功/201创建/400参数错误/404不存在/500服务器错误）
- [ ] 业务错误码设计合理（模块码+错误类型+序号）
- [ ] 错误响应格式统一
- [ ] 错误信息清晰，包含详细信息
- [ ] 参数验证错误返回400，业务错误返回422

## 接口文档
- [ ] 每个接口都有文档
- [ ] 文档包含：接口路径/方法/描述/参数/响应/错误码
- [ ] 提供请求和响应示例
- [ ] 错误码表完整

## 安全性
- [ ] 需要认证的接口有说明
- [ ] 需要权限的接口有说明
- [ ] 敏感数据不在URL中传递
- [ ] 密码等敏感字段不在响应中返回
- [ ] 考虑限流和防刷

## 性能考虑
- [ ] 列表接口有分页（默认20条）
- [ ] 支持字段过滤（只返回需要的字段）
- [ ] 考虑缓存策略（ETag/Last-Modified）
- [ ] 批量操作接口（减少请求次数）

## 🚨 红灯信号
以下任一出现，立即停止：
- [ ] 路径使用动词（如 /getUser）
- [ ] 所有操作都用POST
- [ ] 所有错误都返回200
- [ ] 错误信息不清晰（如 "操作失败"）
- [ ] 没有分页（一次返回所有数据）
- [ ] 响应格式不统一
- [ ] 密码等敏感信息在响应中返回
