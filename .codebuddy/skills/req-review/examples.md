## 示例 1: 用户注册功能审查（完整流程）

### 场景描述

**业务需求**: 为电商平台设计用户注册功能，支持手机号注册、邮箱验证、密码强度校验。

**需求来源**: TAPD 需求文档

**变更 ID**: `2025-11-10-user-registration`

**原始需求**（PRD 片段）:
```
用户注册功能：
1. 用户可以使用手机号注册
2. 用户需要输入密码，密码要求足够强
3. 发送验证码到手机号
4. 用户输入验证码后完成注册
5. 注册成功后自动登录
6. 用户名可以为空，后续可以补充
```

# 需求审查文档

## 1. 概述

**需求名称**: 用户注册功能  
**变更 ID**: 2025-11-10-user-registration  
**需求来源**: TAPD 需求文档

## 2. 功能需求（FR）

### FR1: 手机号注册

**用户故事**:  
作为新用户，我希望使用手机号注册账号，以便快速完成注册。

**描述**:  
用户在注册页面输入手机号和密码，系统发送验证码到用户手机，用户输入验证码后完成注册。

**输入**:
- 手机号（必填）
- 密码（必填）
- 验证码（必填）

**输出**:
- 注册成功：返回用户信息、JWT Token
- 注册失败：返回错误信息

**前置条件**:
- 手机号未被注册

**后置条件**:
- 用户账号创建成功
- 用户已登录

**业务规则**:
- BR1: 手机号格式验证
- BR2: 密码强度验证
- BR3: 验证码有效期

**验收标准**:
- 手机号格式正确
- 密码符合强度要求
- 验证码正确且未过期
- 注册成功后自动登录

**确认项**:
- [ ] 待质量扫描后填充


### FR2: 验证码发送

**用户故事**:  
作为新用户，我希望收到短信验证码，以便验证我的手机号。

**描述**:  
系统向用户手机号发送 6 位数字验证码，验证码有效期 5 分钟。

**输入**:
- 手机号（必填）

**输出**:
- 发送成功：返回成功消息
- 发送失败：返回错误信息

**前置条件**:
- 手机号格式正确
- 手机号未被注册

**后置条件**:
- 验证码发送成功
- 验证码存储到 Redis（TTL=5分钟）

**业务规则**:
- BR4: 验证码发送频率限制

**验收标准**:
- 验证码为 6 位数字
- 验证码有效期 5 分钟
- 同一手机号 60 秒内只能发送一次

**确认项**:
- [ ] 待质量扫描后填充


### FR3: 用户名补充（可选）

**用户故事**:  
作为已注册用户，我希望可以在注册后补充用户名，以便个性化我的账号。

**描述**:  
注册时用户名可以为空，注册成功后用户可以在个人中心补充用户名。

**输入**:
- 用户名（可选）

**输出**:
- 更新成功：返回用户信息
- 更新失败：返回错误信息

**前置条件**:
- 用户已登录
- 用户名未被占用

**后置条件**:
- 用户名更新成功

**业务规则**:
- BR5: 用户名唯一性验证
- BR6: 用户名格式验证

**验收标准**:
- 用户名长度 4-20 个字符
- 用户名只能包含字母、数字、下划线
- 用户名唯一

**确认项**:
- [ ] 待质量扫描后填充


## 3. 业务规则（BR）

### BR1: 手机号格式验证
- 手机号必须是 11 位数字
- 手机号必须以 13/14/15/16/17/18/19 开头

### BR2: 密码强度验证
- 密码长度 8-20 个字符
- 密码必须包含字母和数字

### BR3: 验证码有效期
- 验证码有效期 5 分钟
- 验证码过期后无法使用

### BR4: 验证码发送频率限制
- 同一手机号 60 秒内只能发送一次验证码
- 同一手机号每天最多发送 10 次验证码

### BR5: 用户名唯一性验证
- 用户名在系统中必须唯一
- 用户名不区分大小写

### BR6: 用户名格式验证
- 用户名长度 4-20 个字符
- 用户名只能包含字母、数字、下划线
- 用户名不能以数字开头

## 4. 数据模型（DM）

### User（用户表）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | BIGINT | PRIMARY KEY, AUTO_INCREMENT | 用户 ID |
| phone | VARCHAR(11) | NOT NULL, UNIQUE | 手机号 |
| password | VARCHAR(255) | NOT NULL | 密码（BCrypt 加密） |
| username | VARCHAR(20) | UNIQUE | 用户名 |
| status | TINYINT | NOT NULL, DEFAULT 1 | 状态（1=正常，0=禁用） |
| created_at | DATETIME | NOT NULL | 创建时间 |
| updated_at | DATETIME | NOT NULL | 更新时间 |

**索引**:
- PRIMARY KEY (id)
- UNIQUE KEY uk_phone (phone)
- UNIQUE KEY uk_username (username)

### VerificationCode（验证码表 - Redis）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| phone | STRING | 手机号（Key） |
| code | STRING | 验证码（6 位数字） |
| ttl | INT | 有效期（秒，默认 300） |

## 总结

### 核心要点

1. **结构化建模** - 将原始需求转化为 FR、BR、数据模型、验收标准
2. **6维度扫描** - 系统化执行一致性、完整性、清晰性、准确性、无重复性、可验证性扫描
3. **问题分级** - P0（阻塞性）、P1（重要）、P2（优化）
4. **问题归集** - 按 FR 编号嵌入问题到确认项
5. **风险评估** - 评估整体风险，判断是否可进入设计阶段
6. **迭代审查** - 支持多次审查，标记已澄清问题

### 最佳实践

✅ **问题描述要具体**: 提供清晰的问题描述和建议方案  
✅ **优先级判断要准确**: 遵循 P0/P1/P2 定义标准  
✅ **保留历史记录**: 不删除既有确认项，标记已澄清问题  
✅ **配合需求澄清**: 先澄清再审查，效率更高  

### 常见错误

❌ **忽略合规检查**: 未经要求就生成文档  
❌ **扫描维度不全**: 仅检查部分维度  
❌ **优先级不准确**: 所有问题标 P0  
❌ **删除既有内容**: 更新时删除原确认项
