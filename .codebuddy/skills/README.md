## 🎯 什么是 Skill？

**Skill** 是一个原子性的、可复用的知识单元，每个技能包含：

- ✅ **详细的执行步骤** - 清晰的操作流程
- ✅ **最佳实践** - 经过验证的高效方法
- ✅ **常见错误和解决方案** - 避坑指南
- ✅ **验证清单** - 质量保证检查点
- ✅ **实际示例** - 可参考的真实案例

## 📚 技能分类详览

### 📋 业务分析（6 个）

完整的业务分析工作流，从原始需求到可执行的用户故事和验收标准。

| 技能名称 | 说明 | 触发场景 |
|---------|------|----------|
| **ba-01-business-requirement-extraction** | 从会议记录、访谈、需求文档中提取结构化业务需求，生成 SMART 业务目标和顶层用例 | 需求提取、业务分析、会议记录分析 |
| **ba-02-epic-analysis** | 分析大型 Epic 需求并制定拆解策略，识别用户角色、评估业务价值、选择拆解维度 | Epic 分析、需求拆解、史诗分析 |
| **ba-03-user-story-generation** | 基于 Epic 分析结果生成符合敏捷标准的用户故事列表，遵循"作为...我希望...以便..."格式 | 生成用户故事、敏捷需求、故事编写 |
| **ba-04-invest-evaluation** | 对用户故事进行 INVEST 原则评估和 MoSCoW 优先级排序，识别质量问题并提出改进建议 | INVEST 评估、优先级排序、故事质量检查 |
| **ba-05-acceptance-criteria** | 为高优先级用户故事编写详细的验收标准，使用 Given-When-Then 格式 | 验收标准、AC 编写、BDD 场景 |
| **ba-06-ui-html-prototype** | UI原型设计工作流，包含设计元素分析、交互设计、HTML原型生成三个任务 | UI原型、HTML原型、交互设计 |

### 🎯 需求分析（4 个）

专注于需求的理解、澄清和分解，确保需求质量。

| 技能名称 | 说明 | 触发场景 |
|---------|------|----------|
| **req-review** | 对功能需求进行6维度质量扫描（一致性、完整性、清晰性、准确性、无重复性、可验证性） | 需求文档审查、质量检查 |
| **req-explain** | 将原始需求转化为清晰的功能性需求和验收标准，通过澄清循环消除歧义 | 需求文档编写、需求理解 |
| **req-clarify** | 通过结构化提问识别需求模糊点和缺失信息，每个问题提供2-4个可选方案 | 需求澄清、设计前确认 |
| **req-breakdown** | 将需求文档拆解为可执行的功能任务，确保需求原子化、依赖清晰、可并行分析 | 任务规划、开发拆解 |

### 🔧 技术设计（7 个）

覆盖架构、数据库、API、业务流程等全方位设计能力。

| 技能名称 | 说明 | 技术栈 |
|---------|------|--------|
| **techdesign-01-architecture** | 从系统层面设计整体架构，聚焦架构风格选择、技术选型、模块划分 | 架构设计、系统设计 |
| **techdesign-02-process** | 设计业务流程和状态机，专注流程梳理、状态转移、业务规则 | 业务流程、状态机 |
| **techdesign-03-feature** | 基于需求文档，设计功能的详细用例、输入输出、边界条件和错误处理 | 功能设计、用例设计 |
| **techdesign-04-entity** | 基于领域驱动设计（DDD）进行实体建模和领域设计 | DDD、实体建模 |
| **techdesign-05-database** | 基于业务实体设计规范化的数据库表结构，遵循 MySQL + MyBatis-Plus 最佳实践 | MySQL、MyBatis-Plus |
| **techdesign-06-api** | 设计 RESTful API 接口，构建直观、可扩展、易维护的 API | RESTful、API 设计 |
| **techdesign-07-delivery-planning** | 基于需求分析制定完整的技术方案和实施计划，包含工作量评估、时间规划 | 项目管理、交付计划 |

### 📝 文档工程（10 个）

全面的文档生成能力，覆盖代码分析、索引、清单、项目知识提取、教程生成等多种场景。

| 技能名称 | 说明 | 重点能力 |
|---------|------|----------|
| **doc-code2comment** | 为已有代码编写规范的注释，分析代码逻辑并生成清晰、准确、符合语言规范的注释 | 代码注释、文档化 |
| **doc-analyze-code** | 深度分析代码文件和项目架构，生成结构化的分析报告 | 代码质量、架构分析 |
| **doc-code2req** | 从代码逆向推导需求文档，强制扫描所有文件 + 深度方法调用跟踪 + 业务逻辑提取 | 逆向工程、需求重建 |
| **doc-api-list** | 扫描后台代码生成 API 接口清单文档，包含路径、方法、参数、返回值和使用场景 | API 文档、接口清单 |
| **doc-public-api-list** | 为指定模块生成面向第三方开发者的详细 API 使用文档，确保仅依赖文档即可完成集成 | 第三方集成、API 使用文档 |
| **doc-db-schema** | 基于 MySQL 数据库自动生成完整的表结构文档，包含字段、索引、约束、DDL | 数据库文档、表结构 |
| **doc-git-list** | 分析代码仓库结构，生成完整的代码库索引文档，包括架构分析、业务逻辑映射 | Git 仓库、代码索引 |
| **doc-index** | 生成项目文档索引，扫描并分类所有文档，提供结构化导航和快速查询功能 | 文档导航、索引管理 |
| **doc-tutorial** | 使用四步连续流程为接口、方法及工具生成高质量应用文档 | 技术文档、教程生成 |
| **doc-extract-knowledge** | 从代码项目中提取知识库文档，自动识别场景选择最佳策略（自顶向下/自底向上/按模块） | 项目知识提取、知识库构建 |

## 📂 目录结构

每个 Skill 都遵循统一的结构：

```
skill-name/
├── SKILL.md           # 主文档（必需）- 包含核心流程和详细步骤
├── examples.md        # 示例文档（推荐）- 真实案例展示
├── checklist.md       # 检查清单（推荐）- 检查项确保质量
├── reference.md       # 参考资料（可选）- 深度技术文档
└── templates/         # 模板目录（可选）- 代码模板和配置示例
```

**文档质量标准**：
- ✅ **SKILL.md** - 清晰的执行步骤、触发条件、常见错误
- ✅ **examples.md** - 真实案例，覆盖完整流程、边界情况、错误处理
- ✅ **checklist.md** - 检查项，覆盖每个步骤的详细验证点

## 🎓 最佳实践

### 技能使用建议

1. **业务分析阶段** - 使用 `ba-*` 系列完成从原始需求到用户故事的转化
   - `ba-01` → `ba-02` → `ba-03` → `ba-04` → `ba-05` → `ba-06`

2. **需求阶段** - 使用 `req-*` 系列确保需求质量
   - `req-review` → `req-explain` → `req-clarify` → `req-breakdown`

3. **设计阶段** - 遵循自顶向下的设计流程
   - `techdesign-01-architecture` → `techdesign-04-entity` → `techdesign-05-database` → `techdesign-06-api`

4. **实现阶段** - 利用代码生成和文档工具
   - `code-generation` → `code-review` → `doc-code2comment`

5. **测试阶段** - 建立完整的测试体系
   - `tdd-testing`（统一测试工作流）

6. **安全阶段** - 上线前安全检查
   - `code-security-scan` → 修复问题 → 复查

### 质量保证

每个技能都包含完整的质量保证机制：
- ✅ **examples.md** - 真实案例验证
- ✅ **checklist.md** - 详细检查清单
- ✅ **常见错误** - 避坑指南
- ✅ **最佳实践** - 经过验证的高效方法
