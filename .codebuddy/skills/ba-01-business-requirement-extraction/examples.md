## 简单案例：访客管理系统

### 案例背景

某中型企业（员工500+，每日访客量200+）需要建设访客管理系统：

**组织架构**：
- 1个总部园区，3栋办公楼
- 安保部门负责访客管理

**当前痛点**：
- 纸质登记效率低，访客排队等待时间长
- 安保人员手动核验身份，工作量大
- 访客数据无法统计分析，安全风险难以追溯

### 第1部分：业务目标示例

```markdown
### 业务目标1：提升访客管理效率

- **Specific**: 实现访客预约、审批、签到全流程线上化，取代现有纸质登记方式
- **Measurable**: 
  - 访客预约处理时间从平均30分钟降低到5分钟
  - 安保人员现场确认时间从10分钟降低到2分钟
  - 系统自动审批率达到60%以上
- **Achievable**: 通过微信公众号集成、门禁系统对接、OCR身份证识别技术实现
- **Relevant**: 当前每日访客量200+，人工处理效率低、安全风险高、访客体验差
- **Time-bound**: 2025年Q2上线MVP版本，Q3实现全面推广
- **来源**: （引用自会议记录第1页"目标：提升安全性与便捷性"）
```

### 第2部分：顶层业务用例示例

```markdown
### 业务用例1：管理访客预约申请

- **主要参与者**: 访客（外部来访人员）
- **利益相关者**: 
  - 访客：希望提前预约，避免现场等待
  - 被访人：希望提前知晓访客信息
  - 安保人员：希望减少现场登记工作量
  - 企业管理层：希望掌握访客数据
- **业务目标**: 
  - 提升访客预约效率（处理时间从30分钟降至5分钟）
  - 降低安保工作量（现场确认时间从10分钟降至2分钟）
- **前置条件**: 
  - 访客已获知被访人信息
  - 访客持有有效身份证件
- **后置条件**: 
  - 成功：预约申请已提交，被访人收到审批通知
  - 失败：预约申请被拒绝，访客收到拒绝原因通知
- **主要业务流程**:
  1. 访客通过微信公众号进入预约申请页面
  2. 访客填写个人信息（姓名、手机号、身份证号）
  3. 访客上传证件照片
  4. 访客填写来访信息（来访目的、访问日期、访问时段）
  5. 访客搜索并选择被访人
  6. 系统验证信息完整性
  7. 系统生成预约单号
  8. 系统推送审批通知给被访人
- **扩展流程**:
  - **E1.重复提交**: 系统检测到当天已有预约，显示警告并禁用提交
  - **E2.照片上传失败**: 系统显示错误提示，访客重新上传
  - **E3.被访人不存在**: 系统显示未找到匹配员工，访客重新搜索
- **业务规则**:
  - BR1: 同一访客每天只能提交1次预约申请
  - BR2: 访客照片格式JPG/PNG，大小≤5MB
  - BR3: 访问日期范围：今天～今天+30天
  - BR4: 被访人必须是在职员工
- **量化指标**:
  - 预约提交成功率：≥95%
  - 通知推送成功率：≥98%
- **优先级**: Must Have
- **来源**: （引用自会议记录第1页"访客希望提前预约"）
```

## 复杂案例：全球化员工入职管理平台

### 案例背景

某全球化互联网头部企业（员工10万+，业务覆盖30+国家/地区）需要建设统一的入职管理平台：

**组织架构**：
- 集团总部 → 6大业务群(BG) → 30+业务单元(BU) → 100+区域公司 → 50+海外法律实体
- HR运营模式：三支柱模型（SSC共享服务中心 + COE专家中心 + HRBP业务伙伴）

**当前痛点**：
- 12种员工类型流程各异，缺乏统一管理
- 50+法律实体审批规则不统一，合规风险高
- 年入职量2万+人次，校招季单日峰值1000+，系统性能瓶颈
- 多国劳动法合规要求复杂（中国、GDPR、美国EEOC、日本劳动基准法等）
- 20+核心系统对接，数据孤岛严重

### 第1部分：业务目标示例（企业级）

```markdown
### 业务目标1：建设全球统一入职管理平台

- **Specific**: 
  - 支持12种员工类型差异化入职流程（正式员工-社招/校招、实习生、外包、派遣、返聘、高管、博士后、兼职、董事、海外本地雇员、跨境派遣）
  - 支持6大BG + 30+BU + 50+海外实体的组织架构
  - 实现集团管控与区域自治的平衡（集团定政策、区域定细则）
  - 支持中国、美国、欧盟、日本、新加坡等主要国家/地区的合规要求
- **Measurable**: 
  - 入职准备时间从5天缩短至1天
  - 信息录入错误率从12%降至0.5%以下
  - 校招季批量入职处理能力：1000人/天
  - 规则变更响应时间从2周降至4小时（可配置化）
  - 新员工满意度提升至95%+
- **Achievable**: 
  - 通过规则引擎实现动态流程配置
  - 对接集团统一认证、全球组织架构、薪酬系统、资产管理等20+核心系统
  - 采用多租户架构实现数据隔离
- **Relevant**: 
  - 当前12种用工类型流程混乱，SSC工作量大
  - 海外实体合规风险高，已收到监管警告
  - 校招季系统崩溃影响雇主品牌
- **Time-bound**: 
  - 2025年Q2：国内版本上线（覆盖总部+国内分公司）
  - 2025年Q3：亚太版本上线（新加坡、日本、韩国）
  - 2025年Q4：全球版本上线（欧美）
- **来源**: （引用自集团HRVP访谈"全球化扩张带来的入职管理挑战"）
```

### 第2部分：顶层业务用例示例（企业级）

```markdown
### 业务用例1：全球化多类型员工入职信息采集

- **主要参与者**: 
  - 新员工（正式/校招/实习/返聘/高管/博士后/兼职）
  - 供应商管理员（外包/派遣）
  - 海外本地HR（海外本地雇员/跨境派遣）
- **利益相关者**: 
  - 新员工：希望快速完成信息填写，多语言支持，移动端友好
  - SSC专员：希望系统自动判定员工类型和审批链，减少手动操作
  - HRBP：希望提前了解新员工信息，参与关键审批
  - COE专家：希望政策规则可配置，快速响应业务变化
  - 部门经理：希望快速审批，了解新人背景
  - 法务合规：希望确保合同类型与员工类型匹配，满足各国劳动法
  - 财务部门：希望准确获取薪资账户信息，支持多币种
  - 安全部门：希望涉密岗位有额外审批，背调完整
  - 集团HR：希望全局可视，政策统一
- **业务目标**: 
  - 根据员工类型+组织+国家自动加载差异化表单
  - 一次录入，多处复用（HR系统、财务系统、门禁系统、IT系统）
  - 满足各国数据本地化存储要求
- **前置条件**: 
  - 员工类型已确定（系统自动判定或SSC手动选择）
  - 组织架构信息已同步（BG/BU/区域/实体）
  - 用户已通过身份验证（SSO或临时令牌）
- **后置条件**: 
  - 成功：信息采集完成，触发后续审批流程，数据同步至相关系统
  - 失败：信息校验不通过，提示具体错误原因，记录失败日志
- **主要业务流程**:
  1. 系统根据合同类型、雇佣主体、用工方式、入职国家自动判定员工类型
  2. 系统根据员工类型+入职组织+入职国家加载对应表单模板
  3. 新员工/供应商/海外HR填写基本信息
  4. 系统根据员工类型加载差异化字段：
     - 正式员工-社招：银行账户、紧急联系人、学历信息、背调授权
     - 正式员工-校招：学校信息、导师信息、批次信息、宿舍需求
     - 实习生：学校信息、实习期限、导师信息、三方协议
     - 外包人员：供应商信息、驻场项目、权限范围、服务协议
     - 劳务派遣：派遣公司信息、派遣期限、比例检查
     - 返聘顾问：原工号、退休日期、顾问费标准、劳务合同
     - 高管(VP+)：竞业协议、股权激励、背调加强、保密协议
     - 博士后：科研项目、出站安排、导师信息
     - 兼职员工：工时限制、社保处理、兼职协议
     - 董事/监事：董事会决议、工商变更、聘任协议
     - 海外本地雇员：当地税号、工作签证、当地社保
     - 跨境派遣：双重社保、税务处理、派遣协议
  5. 系统根据入职国家加载合规字段：
     - 中国：身份证、户籍、政治面貌
     - 美国：SSN、I-9验证、EEO信息
     - 欧盟：GDPR同意书、税号
     - 日本：My Number、在留资格
     - 新加坡：NRIC/FIN、工作准证
  6. 系统校验必填项和格式
  7. 系统保存信息并触发审批流程
  8. 系统同步数据至相关下游系统
- **扩展流程**:
  - **E1.自动判定失败**: 系统无法判定员工类型，SSC手动选择并填写原因，记录审计日志
  - **E2.外包人员信息**: 由供应商门户提交，甲方SSC审核，采购参与确认
  - **E3.海外员工**: 由当地HR协助填写，总部合规审核
  - **E4.高管入职**: 加载竞业协议、股权激励字段，触发CEO/董事会审批
  - **E5.批量入职**: 校招季支持Excel批量导入，批量审批
  - **E6.跨BG调动**: 原BG HR发起，新BG HR接收，按新入职处理
- **业务规则**:
  - BR1: 员工类型判定规则（12种类型判定矩阵）
  - BR2: 动态审批链规则（按BG+组织+岗位级别+员工类型+薪资档位+特殊标签）
  - BR3: 敏感信息加密存储（身份证、银行卡、SSN等）
  - BR4: 数据本地化存储（中国数据存中国、欧盟数据存欧盟）
  - BR5: 外包比例限制（部分BU外包不超过30%）
  - BR6: 高管入职需背调完成后才能审批
  - BR7: 海外入职需签证/工作许可验证
- **量化指标**:
  - 表单填写完成率：≥98%
  - 员工类型自动判定准确率：≥99%
  - 信息同步成功率：≥99.9%
  - 批量入职处理能力：1000人/天
  - 表单加载时间：<3秒（全球任意地区）
- **优先级**: Must Have
- **来源**: （引用自集团HRVP访谈"12种员工类型流程差异大，合规风险高"）
```
