# 业务需求提取

从非结构化的原始需求信息中提取结构化业务需求，生成 SMART 业务目标和顶层业务用例清单。

## 工作流位置

```
ba-01 业务需求提取 ← 当前技能
    ↓ 输出：业务目标、顶层用例、非功能需求
ba-02 Epic 分析
    ↓ 输出：拆解策略、用户角色分析、MVP 定义
ba-03 用户故事生成
    ↓ 输出：用户故事列表
ba-04 INVEST 评估
    ↓ 输出：评估报告、优先级排序
ba-05 验收标准
```

**下游使用**: ba-02 将使用本技能输出的顶层用例作为 Epic 分析的输入

## 执行检查清单

复制此清单跟踪进度：

```
任务进度：
- [ ] 步骤1：阅读原始材料，标记关键信息
- [ ] 步骤2：提取业务目标（SMART格式）
- [ ] 步骤3：识别顶层业务用例
- [ ] 步骤4：提取非功能需求
- [ ] 步骤5：识别约束条件和假设依赖
- [ ] 步骤6：分析遗漏风险，生成待确认清单
- [ ] 步骤7：质量验证（可追溯性、SMART符合度）
```

## 输入材料

```
【材料1：原始需求信息】
{粘贴：会议记录、访谈内容、需求文档、邮件往来、用户反馈等}

【材料2：业务背景（可选）】
{粘贴：项目背景、行业特点、现有系统情况、组织架构等}
```

## 核心输出

### 第1部分：业务目标（SMART原则）

每个业务目标包含：

| 要素 | 说明 |
|------|------|
| Specific | 具体要实现什么业务成果 |
| Measurable | 量化指标是什么 |
| Achievable | 基于什么资源和能力可实现 |
| Relevant | 解决什么业务问题 |
| Time-bound | 何时达成，里程碑是什么 |
| 来源 | 引用自原始材料第X页/段落Y |

### 第2部分：顶层业务用例

使用**顶层用例法**描述业务需求（聚焦业务价值，非系统功能细节）：

| 要素 | 说明 |
|------|------|
| 主要参与者 | 谁发起这个业务活动 |
| 利益相关者 | 谁关心这个业务活动的结果 |
| 业务目标 | 要达成什么业务价值 |
| 前置条件 | 执行前必须满足什么条件 |
| 后置条件 | 执行后系统/业务状态如何改变 |
| 主要业务流程 | 参与者→系统的交互序列 |
| 扩展流程 | 异常场景及处理方式 |
| 业务规则 | 必须遵守的业务约束 |
| 量化指标 | 如何衡量成功 |
| 优先级 | Must/Should/Could/Won't Have |
| 来源 | 引用自原始材料 |

**NFR 优先级说明**：
- **High**：系统上线必须满足，不满足则无法投产
- **Medium**：重要但可分阶段实现，影响用户体验
- **Low**：锦上添花，可在后续版本优化

### 第4部分：约束条件与假设依赖

- 技术约束、时间约束、预算约束、合规约束
- 假设及其风险、外部依赖及影响

### 第5部分：需求遗漏风险与补充建议

列出可能遗漏的需求点、理由、建议和确认对象。

## 质量要求

### 可追溯性（强制）

- 每个需求标注来源：`（引用自{文档名称}第{页码}页/{段落}"{关键原文}"）`
- 推断或补充的需求标注：`（基于{行业经验/最佳实践}推断，需与业务方确认）`

### SMART原则（强制）

- 业务目标100%符合SMART五要素
- 避免模糊表达："提升效率"→"处理时间从30分钟降低到5分钟"
- 避免主观评价："性能好"→"接口响应时间P95<3秒"

### 需求描述清晰性（强制）

- 使用"参与者→系统"交互方式描述业务流程
- 避免技术实现细节："使用Redis缓存"→"查询响应时间<2秒"
- 避免过早UI设计："表单有3个输入框"→"填写个人信息（姓名、手机号、身份证号）"

### 优先级评估（MoSCoW方法）

| 优先级 | 业务价值 | 用户影响 |
|--------|----------|----------|
| Must Have | 核心业务流程，没有则无法运营 | 影响所有用户 |
| Should Have | 重要业务场景，影响效率 | 影响部分用户 |
| Could Have | 增值场景，锦上添花 | 影响少数用户 |
| Won't Have | 低价值或不紧急 | 几乎无影响 |

## 信息不足处理

1. **合理推断**: 基于行业最佳实践补充
2. **明确标注**: `（推断，需确认）`
3. **列入待确认清单**: 在第5部分列出
4. **提供确认建议**: 说明需要向谁确认

## 需求冲突标注

```markdown
⚠️ **需求冲突警告**
- **冲突点**: {描述}
- **冲突来源**: 需求A（引用自材料1）vs 需求B（引用自材料2）
- **冲突分析**: {为什么冲突}
- **建议解决方案**: 方案1（优先推荐）/ 方案2
- **确认对象**: {需要谁来决策}
```

## 质量验证循环

完成初稿后，执行以下验证：

1. **SMART 符合度检查**：每个业务目标是否满足 SMART 五要素？
2. **来源追溯检查**：每条需求是否标注了来源？
3. **用例完整性检查**：顶层用例是否包含所有必要要素？

**验证→修正→重复**：如发现问题，修正后重新检查，只有全部通过才输出最终文档。

## 技术协作接口

本技能输出需与以下角色协作确认：

| 协作角色 | 协作内容 | 协作时机 |
|---------|---------|---------|
| 架构师 | NFR 技术可行性评审、架构约束确认 | NFR 提取后 |
| 安全团队 | 安全合规要求确认 | 安全性 NFR 提取后 |
| 运维团队 | 可用性、可观测性要求确认 | 相关 NFR 提取后 |
| 业务方 | 业务目标和优先级确认 | 全部输出后 |
