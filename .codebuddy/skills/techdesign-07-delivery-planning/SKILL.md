# Skill: 交付规划与技术方案 (Delivery Planning)

## 工作流位置

```
techdesign-01 架构设计
    ↓ 输出：架构方案、技术选型
techdesign-02 流程设计（可选）
    ↓ 输出：流程图、状态机
techdesign-03 功能设计
    ↓ 输出：功能规格、用例设计
techdesign-04 实体设计
    ↓ 输出：实体模型、领域模型
techdesign-05 数据库设计 ─┬─ 可并行
techdesign-06 API设计    ─┘
    ↓ 输出：DDL、API文档
techdesign-07 交付规划 ← 当前技能（可选，需要项目管理时使用）
    ↓ 输出：任务清单、里程碑、风险评估
→ 编码实现
```

**上游输入**: 所有设计文档（05-database DDL、06-api API清单等）
**下游使用**: 开发团队将使用本技能输出的任务清单进行编码实现
**何时使用**: 需要项目管理、任务分配、进度跟踪、风险评估
**路径选择**: 参见 [techdesign-01-architecture 路径选择指南](mdc:skills/techdesign-01-architecture/SKILL.md)

> ⚠️ **必读**: [通用规范](mdc:.codebuddy/spec/global/standards/common/index.md) - 项目记忆引用和所有规范要求

## 核心原则（15秒速查）

1. **科学估算** - 三点估算法（乐观/正常/悲观）+ 20%缓冲
2. **合理粒度** - 任务1-3天，可独立验收
3. **明确依赖** - 识别关键路径，区分串行/并行
4. **全面风险** - 技术+进度+人员，每个风险有应对措施
5. **量化标准** - 质量标准可测量（≥80%、<200ms）
6. **敏捷度量** - 跟踪 Velocity、Lead Time 等指标
7. **技术债管理** - 识别、评估、规划偿还

## 目标

制定科学、可执行的项目交付计划和技术方案，确保按时、按质量交付。

**适用场景**:
- 需求分析完成，准备进入设计阶段
- 需要评估工作量和资源投入
- 需要制定项目计划和里程碑
- 架构设计完成，需要拆解任务和评估工作量
- 多个设计方案完成，需要整合并规划实施

**触发词**: "制定交付计划"、"评估工作量"、"项目规划"、"任务规划"、"技术方案"

**不适用场景**:
- 需求不明确 → 先使用 `req-clarify`
- 简单功能（<3天）→ 直接编码
- 紧急Bug修复 → 快速修复流程
- 已有详细设计文档 → 直接实现

### 2. 技术选型和架构设计

**原则**：团队能力 + 需求适配 + 长期维护

```markdown
## 技术选型决策

### 后端框架: Spring Boot 3
- **团队能力匹配**: 80%成员熟悉，降低学习成本
- **需求适配性**: 支持RESTful、JWT、事务，满足所有需求
- **长期可维护性**: 活跃社区，LTS版本，降低风险

### 决策矩阵
| 技术项 | 候选方案 | 最终选择 | 核心理由 |
|--------|---------|---------|---------|
| 后端框架 | Spring Boot 3 / Nest.js | Spring Boot 3 | 团队熟悉 |
| 前端框架 | Vue 3 | Vue 3 | 经验丰富 |
| 数据库 | MySQL / PostgreSQL | MySQL | 性能满足 |
```

**架构模式选择**：
- 单体架构：小型项目，团队 < 5人
- 分层架构：中型项目，团队 5-10人
- 微服务架构：大型项目，团队 > 10人

### 3. 功能模块拆解和复杂度评估

**识别功能模块**:
```
示例：电商系统

功能模块:
├─ M1: 用户认证（P0）- 复杂度: 中等(2x)
├─ M2: 商品管理（P0）- 复杂度: 简单(1x)
├─ M3: 购物车（P0）- 复杂度: 中等(2x)
├─ M4: 订单管理（P0）- 复杂度: 复杂(3x)
└─ M5: 后台管理（P1）- 复杂度: 简单(1x)
```

**复杂度标准**:
- **简单(1x)**: 基础CRUD，无复杂业务逻辑
- **中等(2x)**: 有业务规则，涉及缓存/消息队列
- **复杂(3x)**: 复杂流程，分布式事务，多系统集成

**识别依赖**:
```
依赖关系:
M1 → M2 → M3 → M4
关键路径: M1 → M2 → M3 → M4
```

### 4. 任务拆解和工作量评估

**任务粒度**: 1-3天，可独立验收

**使用三点估算法**:
```
公式: 预期工作量 = (乐观值 + 4×最可能值 + 悲观值) / 6

示例: 后端认证API开发
├─ 乐观值: 1天（成熟框架，无问题）
├─ 最可能值: 2天（处理异常、边界条件）
├─ 悲观值: 4天（遇到技术难点）
└─ 预期值 = (1 + 4×2 + 4) / 6 = 2.5天
```

**工作量汇总**:
```
| 模块 | 前端 | 后端 | 小计 | 复杂度 | 调整后 |
|------|------|------|------|--------|--------|
| M1   | 2天  | 4天  | 6天  | 2x     | 12天   |
| M2   | 3天  | 3天  | 6天  | 1x     | 6天    |
| M3   | 2天  | 3天  | 5天  | 2x     | 10天   |
| M4   | 3天  | 5天  | 8天  | 3x     | 24天   |
| M5   | 2天  | 2天  | 4天  | 1x     | 4天    |
| 缓冲(20%) |  |  |  |  | 11.2天 |
| **总计** | **14.4天** | **20.4天** | **34.8天** | | **67.2天** |
```

**任务拆解示例**:
```markdown
## Phase 1: 基础搭建 (第1周)

### Task 1.1: 项目初始化
- **工作量**: 1人天
- **子任务**:
  - [ ] 创建项目（Vite+Vue 3+TypeScript, Spring Boot 3）
  - [ ] 配置ESLint/Prettier
  - [ ] 配置Git仓库
- **依赖**: 无
- **优先级**: P0（最高）
- **验收标准**:
  - [ ] 项目可正常启动
  - [ ] 代码规范检查通过

### Task 2.1: 用户认证模块
- **工作量**: 3人天
- **子任务**:
  - [ ] 实现登录/注册接口
  - [ ] 实现JWT验证
  - [ ] 实现登录页面
  - [ ] 编写单元测试（覆盖率≥80%）
- **依赖**: Task 1.3（基础框架）
- **优先级**: P0
- **验收标准**:
  - [ ] 用户可正常登录注册
  - [ ] Token验证正常
  - [ ] 单元测试通过
  - [ ] 符合安全要求（密码加密、防SQL注入）
```

### 5. 制定时间表和里程碑

**划分阶段**:
```
阶段0: 项目准备（1周）
└─ 架构设计、环境配置、开发规范

阶段1: 基础搭建（2-3周）
└─ 数据库设计、项目框架、CI/CD

阶段2: 核心功能开发（6-8周）
└─ P0功能模块、前后端联调、单元测试

阶段3: 扩展功能开发（2-4周）
└─ P1功能模块、性能优化、集成测试

阶段4: 测试和优化（2-3周）
└─ 系统测试、性能测试、缺陷修复

阶段5: 部署上线（1周）
└─ 灰度发布、监控配置、正式上线
```

**里程碑计划**:
```
M0: 项目启动（第1周）
├─ 交付: 项目计划、技术方案
└─ 验收: 计划通过评审、环境可用

M1: 基础搭建完成（第4周）
├─ 交付: 数据库、项目框架
└─ 验收: 数据库表结构审查通过、项目可运行

M2: 核心功能完成（第12周）
├─ 交付: P0功能模块、API接口
└─ 验收: 单元测试覆盖率≥80%、代码审查通过

M3: 功能开发完成（第16周）
├─ 交付: 所有功能模块、集成测试报告
└─ 验收: 无P0/P1缺陷、性能指标达标

M4: 测试完成（第19周）
├─ 交付: 测试报告、性能报告
└─ 验收: 系统测试通过、缺陷密度<3/KLOC

M5: 上线发布（第20周）
├─ 交付: 生产环境、部署文档
└─ 验收: 系统可用性>99.5%
```

### 7. 风险识别和应对措施

**技术风险**:
```
风险1: 技术方案复杂度高
├─ 影响: 项目延期、技术债累积
├─ 概率: 中等（30-50%）
├─ 等级: P1（重要）
└─ 应对:
   ├─ 预防: 技术POC验证
   ├─ 应急: 简化方案或引入专家
   └─ 缓冲: 预留2-3天攻关时间

风险2: 第三方依赖不稳定
├─ 影响: 功能中断、集成延期
├─ 概率: 低（10-20%）
├─ 等级: P1（重要）
└─ 应对:
   ├─ 预防: 选择成熟稳定的依赖
   ├─ 应急: 备用方案或自研
   └─ 缓冲: 预留1-2天调整时间
```

**进度风险**:
```
风险1: 需求变更频繁
├─ 影响: 项目延期、成本增加
├─ 概率: 高（50-70%）
├─ 等级: P0（严重）
└─ 应对:
   ├─ 预防: 需求冻结机制、变更流程
   ├─ 应急: 优先级调整、功能裁剪
   └─ 缓冲: 预留3-5天应对变更

风险2: 工作量低估
├─ 影响: 项目延期、加班增加
├─ 概率: 中等（30-50%）
├─ 等级: P1（重要）
└─ 应对:
   ├─ 预防: 三点估算法、参考历史数据
   ├─ 应急: 增加人力、调整优先级
   └─ 缓冲: 已预留20%
```

**质量标准**:
```
代码质量:
├─ 测试覆盖率: ≥80%（关键模块≥90%）
├─ Linter检查: 零错误
├─ 代码审查: ≥2人批准
└─ 复杂度: 圈复杂度<10

性能质量:
├─ API响应时间: P95<200ms
├─ 页面加载时间: <2s（首屏）
├─ 数据库查询: <100ms
└─ 并发支持: ≥100用户

安全质量:
├─ 认证: JWT或Session，支持刷新
├─ 授权: 基于RBAC，细粒度控制
├─ 数据保护: 敏感数据加密
└─ 输入验证: 防SQL注入、XSS
```

## 最佳实践

### 1. 科学估算

✅ **推荐**:
```
使用三点估算法:
预期 = (乐观 + 4×正常 + 悲观) / 6

参考历史数据:
- 建立团队估算基线
- 根据实际调整模型

预留缓冲:
- 总工作量 × 15-20%
- 高风险任务预留更多
```

❌ **避免**:
- 拍脑袋估算
- 满打满算，不预留缓冲

### 2. 合理粒度

✅ **推荐**:
```
任务粒度: 1-3天
独立可验证: 明确验收标准
清晰依赖: 明确串行/并行

示例:
T1: 数据库设计（0.5天）
T2: 后端API开发（2天）
T3: 前端页面开发（1.5天）
T4: 前后端联调（1天）
```

❌ **避免**:
- 过粗（>3天）- 无法跟踪
- 过细（<0.5天）- 管理成本高

### 3. 明确验收

✅ **推荐**:
```
M2: 核心功能完成
├─ 交付物: P0功能、API接口
├─ 验收标准:
│  ├─ 所有P0功能可用（100%）
│  ├─ 单元测试覆盖率≥80%
│  └─ 代码审查通过（≥2人）
└─ 关键指标: 0个P0缺陷
```

❌ **避免**:
- 验收标准模糊（"功能基本可用"）

## 常见问题

### Q: 如何确保估算准确？

**A**: 
1. 使用三点估算法，避免单点估算
2. 参考历史项目数据，建立估算基线
3. 预留15-20%缓冲
4. 定期检视实际vs计划，调整模型

### Q: 缓冲时间怎么预留？

**A**:
```
计算: 总工作量 × 15-20%
分配: 均匀分配或集中在高风险阶段

示例:
总工作量30人天，缓冲20% = 6人天
实际周期 = 30 + 6 = 36人天
```

### Q: 如何处理需求变更？

**A**:
1. 评估变更影响（工作量、时间、资源）
2. 优先级排序（必须做vs最好做）
3. 调整交付计划，明确成本
4. 获得关键方认可
5. 更新项目计划

## 敏捷指标 (Agile Metrics)

### 核心指标

| 指标 | 定义 | 计算方式 | 健康值 |
|------|------|---------|--------|
| **Velocity** | 团队每迭代完成的故事点 | 已完成故事点总和 | 稳定波动 < 20% |
| **Lead Time** | 从需求提出到上线的时间 | 上线时间 - 需求创建时间 | < 2周 |
| **Cycle Time** | 从开始开发到上线的时间 | 上线时间 - 开发开始时间 | < 1周 |
| **Throughput** | 单位时间完成的任务数 | 完成任务数 / 时间 | 稳定或上升 |
| **WIP** | 进行中的工作数量 | 当前进行中任务数 | < 团队人数 × 1.5 |

## OKR 对齐

### 什么是 OKR

OKR (Objectives and Key Results) 是目标与关键结果管理方法，确保项目目标与组织战略对齐。

### OKR 模板

```markdown
## 项目 OKR

### Objective 1: 提升用户体验
**关键结果**:
- KR1: 页面加载时间 P95 < 2秒（当前 3.5秒）
- KR2: 用户满意度评分 ≥ 4.5/5（当前 4.0）
- KR3: 关键流程完成率 ≥ 95%（当前 88%）

### Objective 2: 提高系统稳定性
**关键结果**:
- KR1: 系统可用性 ≥ 99.9%（当前 99.5%）
- KR2: P0 故障数 ≤ 1次/月（当前 3次）
- KR3: 故障恢复时间 < 15分钟（当前 30分钟）

### Objective 3: 加速交付效率
**关键结果**:
- KR1: Lead Time < 2周（当前 3周）
- KR2: 部署频率 ≥ 2次/周（当前 1次/周）
- KR3: 变更失败率 < 5%（当前 10%）
```

### 任务与 OKR 对齐

```markdown
## 任务 OKR 对齐表

| 任务 | 对齐的 OKR | 贡献度 |
|------|-----------|--------|
| 前端性能优化 | O1-KR1 | 高 |
| 缓存架构升级 | O1-KR1, O2-KR1 | 高 |
| 监控告警完善 | O2-KR2, O2-KR3 | 中 |
| CI/CD 优化 | O3-KR1, O3-KR2 | 高 |
| 代码重构 | O3-KR3 | 中 |

### 优先级调整
根据 OKR 贡献度调整任务优先级:
- 高贡献度任务优先安排
- 与 OKR 无关的任务降低优先级或移除
```

## 技术债管理

### 什么是技术债

技术债是为了短期交付速度而做出的技术妥协，需要在未来偿还。

### 技术债分类

| 类型 | 说明 | 示例 |
|------|------|------|
| **设计债** | 架构或设计不合理 | 单体应用需要拆分 |
| **代码债** | 代码质量问题 | 重复代码、复杂度高 |
| **测试债** | 测试覆盖不足 | 缺少单元测试 |
| **文档债** | 文档缺失或过时 | API 文档不完整 |
| **依赖债** | 依赖版本过旧 | Spring Boot 2.x 未升级 |
| **基础设施债** | 运维工具落后 | 手动部署 |

### 技术债评估

```markdown
## 技术债登记表

| ID | 类型 | 描述 | 影响 | 利息 | 偿还成本 | 优先级 |
|----|------|------|------|------|---------|--------|
| TD-001 | 代码债 | 订单模块重复代码多 | 维护困难 | 中 | 3人天 | P1 |
| TD-002 | 测试债 | 支付模块无单测 | 上线风险高 | 高 | 5人天 | P0 |
| TD-003 | 依赖债 | Log4j 版本过旧 | 安全漏洞 | 高 | 1人天 | P0 |
| TD-004 | 设计债 | 用户服务耦合严重 | 扩展困难 | 中 | 10人天 | P2 |

### 评估维度
- **影响**: 对系统的负面影响程度
- **利息**: 不偿还导致的持续成本（每迭代增加的维护成本）
- **偿还成本**: 修复所需的工作量
- **优先级**: P0(紧急) > P1(重要) > P2(一般) > P3(低)
```

### 技术债偿还策略

```markdown
## 偿还策略

### 策略1: 固定比例
每个迭代预留 15-20% 时间偿还技术债

Sprint 容量: 30 故事点
技术债预留: 30 × 20% = 6 故事点

### 策略2: 专项迭代
每 4 个迭代安排 1 个技术债专项迭代

Q1 计划:
- Sprint 1-3: 功能开发
- Sprint 4: 技术债偿还

### 策略3: 搭车偿还
在功能开发时顺便偿还相关技术债

任务: 订单查询优化
搭车偿还: 重构订单模块重复代码

### 策略4: 紧急偿还
安全漏洞、严重性能问题立即偿还

触发条件:
- 安全漏洞（CVE 评分 ≥ 7）
- P0 故障根因
- 严重性能问题
```
