# 任务规划参考

## 技术选型三要素

### 1. 团队能力匹配
- 评估团队对技术的熟悉程度
- 考虑学习成本和培训时间
- 优先选择团队已掌握的技术

### 2. 需求适配性
- 技术能否满足所有功能需求
- 性能、安全、扩展性是否达标
- 是否支持未来演进需求

### 3. 长期可维护性
- 社区活跃度和生态完善度
- 是否有长期支持版本（LTS）
- 团队能否持续维护和升级

## 任务拆解原则

### 粒度控制
- **推荐**：1-3天
- **可接受**：0.5-5天
- **避免**：>5天（过粗），<0.5天（过细）

### 验收标准
```markdown
# ✅ 好的验收标准
- [ ] 用户可正常登录注册
- [ ] Token验证正常
- [ ] 单元测试覆盖率≥80%
- [ ] API响应时间<200ms

# ❌ 差的验收标准
- [ ] 完成开发
- [ ] 功能可用
```

### 依赖管理
```markdown
# 标注依赖关系
Task 2.1: 用户认证
- **依赖**: Task 1.3（基础框架）

Task 2.2: 业务模块A
- **依赖**: Task 2.1（用户认证）

# 识别关键路径
关键路径: Task 1.1 → Task 1.3 → Task 2.1 → Task 2.2
```

## 工作量评估方法

### 三点估算法
```
公式: 期望工作量 = (乐观值 + 4×正常值 + 悲观值) / 6

示例:
- 乐观值: 2人天（一切顺利）
- 正常值: 3人天（正常情况）
- 悲观值: 5人天（遇到问题）
- 期望: (2 + 4×3 + 5) / 6 = 3.2人天
- 最终: 3人天
```

### 风险缓冲
```
推荐: 总工作量的20%

示例:
- 开发: 20人天
- 测试: 5人天
- 小计: 25人天
- 风险缓冲: 5人天（20%）
- 总计: 30人天
```

## 优先级定义

| 级别 | 含义 | 示例 |
|------|------|------|
| P0 | 最高 | 核心功能、关键路径任务 |
| P1 | 高 | 重要功能、依赖任务 |
| P2 | 中 | 辅助功能、优化任务 |
| P3 | 低 | 可选功能、未来规划 |

## 里程碑规划

### 定义要素
- **时间节点**：具体日期或周数
- **交付物**：明确的输出
- **验收标准**：可量化的标准
- **关键路径**：是否在关键路径上

### 示例
```markdown
| 里程碑 | 时间 | 交付物 | 验收标准 |
|--------|------|--------|---------|
| M1: 基础完成 | 第1周末 | 项目框架+数据库 | 项目可运行 |
| M2: 功能完成 | 第3周末 | 主要功能 | 功能可用+测试通过 |
| M3: 上线发布 | 第5周末 | 生产环境 | 稳定运行 |
```

## 风险管理

### 风险评估要素
- **影响**：高/中/低
- **概率**：高/中/低
- **应对措施**：具体行动
- **缓冲时间**：量化的时间

### 常见风险
```markdown
| 风险项 | 影响 | 概率 | 应对措施 |
|--------|------|------|---------|
| 需求变更 | 高 | 中 | 敏捷迭代，预留缓冲 |
| 技术难点 | 中 | 低 | 提前技术预研 |
| 人员变动 | 高 | 低 | 知识共享，文档完善 |
| 第三方依赖 | 中 | 中 | 提前对接测试 |
| 性能问题 | 中 | 中 | 性能测试，提前优化 |
```
