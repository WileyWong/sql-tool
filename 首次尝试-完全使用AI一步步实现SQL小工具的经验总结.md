# 首次尝试 — 完全使用 AI 一步步实现 SQL 小工具的经验总结

## 写在前面

市面上好用的 SQL 客户端工具要收费（DataGrip、Navicat），不收费的又总有各种不顺手的地方。既然现在 AI 写代码已经这么强了，何不自己按自己的使用习惯，从零做一个？

这篇文章记录的就是这个过程：**一个不写前端代码的人，如何完全依靠 AI（主要是 CodeBuddy / Cursor 等 AI 编程工具），从零实现一个功能完整的 SQL 客户端桌面工具**。

项目从 2025 年 12 月 19 日开始，到 2026 年 3 月，经历了 20 多次需求迭代。整个过程中代码全部由 AI 生成，人做的事情只有三件：**写需求、验收测试、提 bug**。

这篇文章不是技术教程，而是经验分享——**怎么跟 AI 协作才能真正把一个完整的桌面工具做出来**。

---

## 最终做出来了什么

一个基于 Electron + Vue 3 + TypeScript 的 SQL 客户端桌面工具，主要能力：

- **双数据库支持**：MySQL 和 SQL Server
- **SQL 编辑器**：基于 Monaco Editor，支持自动补全、语法检查（红色波浪线）、代码格式化（Shift+Alt+F）、悬浮提示（表结构/字段信息/函数签名）
- **查询执行**：批量执行、停止执行、执行计划可视化（流程图 + 表格）
- **数据操作**：结果集内直接增删改，SQL 预览确认，事务提交
- **结果展示**：虚拟滚动支持万行数据、列宽自适应、单元格查看器（JSON/XML 格式化）、导出 CSV/JSON/XLSX
- **连接管理**：AES 加密存储、连接树浏览、右键菜单操作
- **国际化**：简体中文、繁体中文、英文三语支持
- **桌面特性**：Electron 原生菜单、标签页拖拽排序、最近文件记录

技术栈：Electron + Vue 3 + TypeScript + Pinia + Monaco Editor + Element Plus + mysql2 + mssql

## 第一阶段：从零到能跑（2025-12-19）—— 先把东西做出来

### 怎么开始的

第一步是写需求文档。当时的想法很简单：把我想要的功能一条条列出来，扔给 AI 让它做。

初始需求文档的写法非常朴素，就是按 story 拆解的方式，把功能点一条条列出来：

```
## story拆解

### 1. 首次打开此工具
1. 数据库连接树形展示 区域提示没有数据，并且增加新建连接的功能

### 2. 连接数据库
1. 弹出连接数据库弹窗
2. 必填字段：名称、服务地址（host:port格式）、用户名、密码
3. 保存后，使用本机MAC地址 + AES对称加密...

### 3. 在sql编辑区编写sql
#### 3.1 自动联想
- 输入select * from 提示所有可用表、视图、函数
- 输入select * from t1 where t1. 提示所有可用字段、函数
...
```

没有什么模板，没有格式规范，就是把脑子里想要的功能用大白话写出来。但有一点做对了：**给了 UI 原型**。提前让 AI 画了两个 HTML 页面——主界面原型和空状态界面原型。做法很简单：随手写几句描述（比如"左边是连接树，右边是编辑器，下面是结果区"），让 AI 生成一个纯 HTML 的静态原型页面，打开看看觉得不满意的地方，再让 AI 改，几轮下来就有了一个还不错的界面参考。有了这个可视化的原型，后续 AI 在实现界面时就能直观理解布局应该长什么样，比纯文字描述有效得多。

### AI 做了什么

拿到需求文档后，AI 的工作流程是这样的：

1. **需求澄清**：AI 主动提了 10 个问题，比如"密码是否必填"、"多 SQL 执行策略是什么"、"停止执行要不要回滚"。每个问题给出选项，我选一个就行。
2. **需求评审**：AI 把需求整理成 FR1-FR10 共 10 个功能需求，每个都有用户故事、验收标准、业务规则，还给出了风险评估。
3. **架构设计**：AI 输出了完整的系统架构，包括模块划分、技术选型、非功能需求。
4. **功能设计**：6 大模块、42 页的详细设计文档，每个用例的输入输出、异常处理、错误码都有。
5. **编码实现**：AI 按设计文档一步步写代码。

### 第一阶段的经验

**做对了的事：**
- 给了 UI 原型，让 AI 直观理解界面
- 让 AI 先做需求澄清和评审，而不是直接写代码
- 功能点描述尽量具体，比如"联想场景"列了 10 多种具体情况

**踩的坑：**
- 需求描述有些地方太模糊，AI 会过度设计。比如错误码体系、重试机制，其实一个小工具根本用不到
- 一次性给的功能太多，AI 的上下文窗口有限，后面的实现质量会下降
- 有些功能的边界没说清楚，AI 会按自己的理解来，最后返工

**核心体会：你给 AI 的需求越清晰，它输出的质量越高。模糊的需求 = 反复返工。**

## 第二阶段：把它变好用（RC-001 ~ RC-010）—— 小步迭代的开始

### 从"能用"到"好用"

第一版做出来后能跑，但很多细节不对。于是进入了持续迭代阶段，通过一个个小需求变更来打磨。

这一阶段一共提了 10 次需求变更（RC-001 到 RC-010），覆盖了：
- 双击切换数据库、单元格编辑、导出功能（RC-001）
- 最近文件菜单、关闭未保存提示（RC-002）
- 表管理对话框（RC-003）
- 建表/改表/删表（RC-004）
- 编辑器结果独立存储（RC-005）
- 六个 bug 修复与优化（RC-006）
- SQL 字符串配色、日期格式化、单元格查看器（RC-007）
- MySQL 函数外置配置（RC-008）
- ORDER BY 补全、Excel 导出（RC-009）
- 编辑状态修复、BIT 类型显示（RC-010）

### 需求文档格式的演变——这是关键转折点

这个阶段最值得说的不是功能本身，而是**需求文档的写法在这个阶段发生了质变**。

#### 最早期（RC-001）：纯自由文本，想到哪写到哪

```markdown
**这是需求变更**

时间：2025年12月22日

在原有功能基础上，实现本文档中提到的需求变更

**1.左侧数据库选择组件**
1. 双击数据库名，将自动切换当前查询连接的数据库
2. 当前选中的数据库和服务器，在CSS上做下区分

**2.查询结果**
1. 如果当前查询为单表，存在主键，双击单元格，可以修改值...
```

特点：没有编号、没有优先级、没有验收标准、没有影响分析。就是一段话。AI 拿到这种需求也能做，但容易遗漏边界情况，也没法追溯。

#### 中期（RC-007）：开始用模板，但只填关键部分

到 RC-006 左右，意识到每次需求变更后 AI 的理解经常有偏差，于是让 AI 生成了一个需求变更模板，包含基本信息、变更概述、变更背景、功能需求表格、输入输出示例、影响分析、测试用例等章节。

从 RC-007 开始，需求文档就按模板来写了。但这时候人并不会把模板每一项都填满，而是**只填自己确定的部分**——功能需求和测试用例，其他的留给 AI 在需求评审阶段补充：

```markdown
# 需求变更单

## 基本信息
| 项目 | 内容 |
|------|------|
| 变更编号 | RC-007 |
| 变更日期 | 2026-01-15 |
| 优先级 | P1-高 |

## 1. 变更内容
### 1.1 功能需求
| 序号 | 功能点 | 描述 | 验收标准 |
|------|--------|------|----------|
| 1 | SQL编辑器SQL语法样式 | 字符串现在是大红色，改成橙色，颜色值：#f5b801 | |
| 2 | DATETIME(fsp)类型支持 | DATETIME可以指定精度，现在无法实现 | |
| 3 | 结果表格日期格式 | DATE显示成yyyy-MM-dd... | |
| 4 | 单元格查看弹出层 | 右键点击单元格，显示"查看"菜单... | |

## 2. 测试用例
| 用例ID | 场景 | 输入 | 期望结果 |
|--------|------|------|----------|
| TC01 | ... | ... | ... |
```

特点：有了结构，AI 能更准确地理解。但人写的部分还是偏简单，主要依赖 AI 自己展开。

### 这一阶段的关键经验

1. **小步快跑**：每次只改 3-5 个功能点，不要一次塞太多。AI 的上下文有限，小批次的质量远高于大批次。
2. **建立模板很重要**：有了统一格式后，AI 的理解准确率明显提升，也方便后续回溯。
3. **测试用例要人写**：AI 生成的测试用例往往覆盖不到真实使用中的边界情况，人写的测试用例更贴合实际。
4. **Bug 可以攒一批一起改**：RC-006 就是攒了 6 个 bug 一起改的，效率比一个个改高。

## 第三阶段：越来越熟练（RC-011 ~ RC-019）—— 建立完整协作流程

### 从"能用模板"到"人机分工明确"

到这个阶段，与 AI 的协作模式已经形成了一套稳定的流程：

```
人写需求（按模板，重点写"变更内容"和"交互设计"）
    ↓
AI 做需求评审 → 输出确认项和风险评估
    ↓
人确认
    ↓
AI 做需求澄清 → 提出模糊点，给出选项
    ↓
人逐条选择
    ↓
AI 做技术设计 → 输出完整的架构/功能设计文档
    ↓
AI 编码实现
    ↓
人验收测试 → 提 bug → AI 修复
```

**这套流程让AI做需求评审、做设计出了完整的文档，在编码实现的时候可以根据文档去做，如果因为上下文长度的原因忘记了原始需求，可以再回去看文档。本质上和人的开发流程是一样的。**

**这时候人和 AI 的分工已经很清晰了：**
- **人写**：变更概述、变更背景、功能需求描述、交互设计（ASCII 图）、输入输出示例、测试用例、需求确认记录
- **AI 补**：需求评审（检查完整性、一致性）、需求澄清（找出模糊点）、技术方案设计、编码实现

#### 最新期（RC-019/RC-020）：格式更精炼，重点更突出

到 RC-019、RC-020 时，需求文档不再严格套模板的每一个章节，而是**按需裁剪**：

- 大需求（RC-019 有 9 个功能点）：写完整的需求详情 + 交互设计 + 验收标准，然后 AI 做完整的需求评审、澄清、功能设计
- 中等需求（RC-020 创建/删除数据库）：重点写交互流程、SQL 生成规则、安全限制，不用 AI 补太多

```markdown
# RC-020 新增/删除数据库功能

### 2.1 创建数据库
**入口：** 右键点击连接节点/数据库节点

**字符集与排序规则的数据来源（动态查询）：**
| 数据项 | MySQL 查询语句 | SQL Server 查询语句 |
|--------|---------------|-------------------|
| 字符集列表 | `SHOW CHARACTER SET` | ❌ 不适用 |
| 排序规则列表 | `SHOW COLLATION WHERE Charset = '...'` | `SELECT name FROM sys.fn_helpcollations()` |

**删除前 Session 清理：**
- 执行 DROP DATABASE 前，遍历所有编辑器 Tab，找到同时满足以下条件的 Tab：
  - 使用的 connectionId 与当前连接相同
  - 使用的数据库名与待删除数据库相同
- 对这些 Tab 调用 session.destroy(tabId, connectionId) 销毁其 session
```

**特点：人已经知道哪些信息 AI 需要、哪些不需要。不多写废话，但关键的技术约束（比如 Session 清理逻辑、SQL 生成规则）一定会写清楚。**

### workspace 目录：给 AI 留下"工作记忆"

这个阶段还有一个重要实践：**建立了 workspace 目录作为"项目记忆"**。

每个需求变更完成后，相关的设计文档、实现记录都归档到按 RC 编号命名的目录下。这样做的好处是：
- AI 在处理新需求时，可以参考历史设计文档，理解项目上下文
- 需求变更有迹可循，不会出现"这个功能当时为什么这么设计"的困惑
- 代码 review 有据可查

目录结构的演变也反映了流程的成熟：

# 早期——比较随意

## 第四阶段：多数据库支持（SQL Server / T-SQL）—— 最大的一次挑战

### 为什么这次不一样

前面所有的迭代都是在 MySQL 单数据库的框架内加功能，但支持 SQL Server 意味着**整个架构要变**：
- 数据库驱动层要抽象（IDatabaseDriver 接口 + DriverFactory）
- SQL 解析器要换（sql-parser-cst 不支持 T-SQL，改为 node-sql-parser）
- 连接模型要变（MySQL 是单连接，SQL Server 是连接池）
- 每个编辑器 Tab 要独立 Session（会话隔离）
- 执行计划要重做（MySQL 是 JSON，SQL Server 是 SHOWPLAN_XML）
- 表管理对话框要拆分（MySQL 和 SQL Server 的建表语法完全不同）

### 这次的需求和设计文档

这次一共产出了多份设计文档，按顺序：

1. **SQL Server 技术设计文档**（53 KB）：最核心的技术设计文档，涵盖数据库抽象层、驱动实现、IPC 适配、前端改造、向后兼容等
2. **编辑器独占连接改造设计**（26 KB）：每个编辑器 Tab 独立 Session 的改造方案
3. **SQL 解析器迁移方案**（7 个文档）：SQL 解析器从 sql-parser-cst 迁移到 node-sql-parser 的分阶段方案
4. **执行计划组件重做**（27 KB）：适配 MySQL JSON 和 SQL Server XML 两种执行计划格式
5. **表管理对话框拆分适配**（18 KB）：MySQL 和 SQL Server 建表语法差异的 UI 适配

### 如何让 AI 完成大规模重构

关键经验：**把大任务拆成独立的小阶段，每个阶段有明确的输入输出和验收标准**。

以解析器迁移为例，拆成了 7 个阶段：

```
01-overview.md         → 整体迁移方案、风险评估
02-dialect-mapping.md  → 方言映射函数
03-diagnostic-provider.md → 语法检查适配
04-completion-provider.md → 自动补全适配
05-sql-parser-service.md  → 核心解析服务替换（最关键）
06-formatting-provider.md → 格式化适配
07-cleanup-and-testing.md → 清理旧依赖 + 测试清单
```

每个文档都写清楚了：要改哪些文件、删多少行、加多少行、有什么风险。AI 按文档一个阶段一个阶段实现，每个阶段完成后验证再进入下一个。

### 这一阶段的体会

- **大重构一定要先出完整设计文档**：不能边做边想，AI 在没有全局视图的情况下做重构，会把代码改得一团糟
- **分阶段实施，每阶段可验证**：中间状态也要能编译通过、跑得起来
- **向后兼容要提前想好**：比如历史连接数据没有 type 字段怎么办，设计文档里就写了 migrateConnections 的逻辑

## 代码质量：让 AI 审自己的代码

在 RC-010 完成后，做了一次代码 review。让 AI 对自己写的代码做审查，结果还挺有意思：

**第一次审查：79 分（B 级）**
- 主要问题：SqlEditor.vue 超过 1000 行、ConnectionTree.vue 878 行、缺少错误边界
- 49 个文件，10 个问题（0 Critical、3 High、5 Medium、2 Low）

然后让 AI 按审查建议重构，重构完再审：

**第二次审查：88 分（B+ 级）**
- SqlEditor.vue 从 ~650 行降到 ~465 行
- 提取了 useLanguageServer、useEditorModel、useTreeFilter 三个 composable
- 只剩 3 个 P2 和 4 个 P3 问题

**让 AI 审自己的代码是个不错的实践**。虽然它不会发现所有问题（比如业务逻辑上的错误），但代码结构、命名规范、性能隐患这些它审得很到位。

## 需求文档编写格式的演变总结

这是整个项目中最值得单独说的一件事。需求文档的写法变化，直接反映了"人对 AI 写代码的理解"在不断加深：

### 第一阶段：纯人写，随意文本

```
**这是需求变更**
时间：2025年12月22日

**1.左侧数据库选择组件**
1. 双击数据库名，将自动切换当前查询连接的数据库
```

- 没结构、没模板、没验收标准
- AI 能做，但理解偏差大，经常需要返工
- 优点：快，门槛低

### 第二阶段：建立模板，人填框架

让 AI 生成了一个需求变更模板，包含完整章节：基本信息、变更概述、变更背景、功能需求表格、输入输出示例、影响分析、测试用例。

人开始按模板写，但只填自己确定的关键部分（功能需求 + 测试用例），其他章节留空或省略。

- AI 理解准确率提升，减少了返工
- 但对于复杂功能，信息量还是不够

### 第三阶段：人写交互设计 + ASCII 图 + 示例，AI 补充技术方案

人意识到**交互细节是 AI 最容易理解偏差的地方**，开始在需求文档中画 ASCII 界面图、写具体的操作步骤、给出 SQL 示例。

```markdown
**二次确认对话框：**
┌─────────────────────────────────────┐
│  ⚠️ 确认删除                         │
│  即将执行以下 SQL 语句：              │
│  │ DELETE FROM users WHERE id = 1;  │
│  ⚠️ 删除后不可恢复，是否继续？        │
│  [取消]          [确认删除]          │
└─────────────────────────────────────┘
```

同时，**让 AI 做需求评审和澄清**成为标准流程。AI 会找出需求中的模糊点，以选择题的形式让人确认。

- 这时候的需求文档质量已经很高，AI 几乎不会理解偏差
- 人写需求的效率也提高了：知道哪些要写、哪些不用写

### 第四阶段：按需裁剪，重点突出

不再死板套模板。简单需求就写几行，复杂需求按模板展开。人已经知道**AI 需要什么信息才能正确实现**。

关键心得：**需求文档不是写给人看的——是写给 AI 看的。AI 需要的是：明确的功能描述、具体的交互行为、边界条件、输入输出示例。它不需要你写"变更原因"或"项目背景"。**

## 经验总结与踩坑记录

### 做对了的事

1. **需求是灵魂**：给 AI 的需求越精确、越具体，输出越好。模糊的需求等于无限返工。
2. **小步快跑**：每次 3-5 个功能点，做完验收再下一批。别一次扔 20 个功能点给 AI。
3. **建立模板**：统一的需求格式减少了沟通成本，也方便回溯。
4. **让 AI 做全流程**：需求澄清 → 需求评审 → 技术设计 → 编码 → Code Review，每一步都让 AI 产出文档，这些文档就是 AI 的"工作记忆"。
5. **给 UI 原型**：不管是 HTML 原型还是 ASCII 图，视觉化的信息比纯文字描述有效 10 倍。
6. **归档工作记录**：workspace 目录按编号归档，AI 可以随时回看历史设计。
7. **大任务拆小阶段**：每阶段可独立验证，出问题容易定位。

### 踩过的坑

1. **AI 过度设计**：给一个小功能，AI 可能给你设计出企业级架构。要明确告诉它范围和约束。
2. **上下文丢失**：长对话后 AI 会忘记前面的需求。解决办法是把关键信息写在文档里，让 AI 随时可以回看。
3. **大文件修改容易出错**：AI 改 1000 行的文件时，经常改错位置或漏改。这也是为什么后来做了代码拆分，把大组件拆成 composable。
4. **不要完全信任 AI 的测试用例**：AI 生成的测试用例覆盖的是"常规路径"，真实使用中的边界情况还是要人来想。
5. **AI 写的 CSS 经常有问题**：布局、间距、颜色这些细节，AI 很难一次做对，经常需要微调。
6. **并行改多个文件时容易冲突**：AI 可能在改 A 文件时假设 B 文件的接口，但 B 文件的实现和假设不一致。分阶段实施可以缓解这个问题。

### 给想尝试的人的建议

1. **先从小工具开始**：别一上来就让 AI 做大系统。先做个几百行的小工具，摸清 AI 的能力边界。
2. **花时间写好需求**：这不是浪费时间。好的需求文档能节省 10 倍的调试时间。
3. **建立自己的协作流程**：找到适合自己的"需求 → 评审 → 设计 → 编码 → 验收"节奏。
4. **保留所有过程文档**：这些文档不仅是给 AI 看的，也是给未来的自己看的。
5. **接受不完美**：AI 写的代码不会完美，但它能把 80% 的工作量完成。剩下 20% 的微调和 bug 修复，通过迭代来解决。
6. **把 AI 当成一个"能力很强但需要明确指令的初级开发者"**：它什么都能做，但你得告诉它具体要做什么。

## 写在最后

这个项目从第一行需求到现在，所有代码都是 AI 写的。它能做到什么程度？说实话，比我预期的好很多。一个完整的、支持双数据库的、有国际化的桌面 SQL 客户端，功能对标商业工具的 80%，这在以前是不可想象的。

从"随便写两句话扔给 AI"到"建立完整的需求模板和协作流程"，这个过程本身就是对"如何使用 AI"的学习曲线。希望这篇记录对想尝试 AI 编程的人有所帮助。
